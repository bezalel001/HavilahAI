1. Problem Framing and Product Hypothesis

Input modality mismatch: students store knowledge in long documents, but revise in short bursts.

Cognitive bottleneck: dense language and weak retrieval practice reduce retention.

Hypothesis: AI-generated micro-learning assets + low-latency interactions improve consistency and outcomes.

Core claim:one upload can be transformed into multiple learning formats aligned with Gen Z usage patterns.

4. LLM Strategy: OpenAI + Anthropic

Provider Use

OpenAI is primary for content planning and quiz/asset generation.

Anthropic support is present for selected learning asset flows.

Fallback logic exists where multi-provider service wrappers are used.

Why this design

Operational resilience across provider outages/rate limits.

Model-choice flexibility per task quality/cost profile.

Separation of provider clients from route/controller logic.

Model layer5 / 18

5. Upload and Extraction Pipeline

Client uploads file (camera/image/document).

Backend validates content type and size.

Storage service writes to local or MinIO-compatible backend.

Extractor routes by type: image OCR, PDF parser, DOCX parser, plain text decode.

Normalized text is passed into LLM asset generation stage.

This decouples ingestion from AI generation, enabling independent improvements per stage.

8. Why Async Queueing Was Added

Sync-only limitations

Long request latency

Higher timeout risk

Poor scalability under concurrency

Queue-based flow

API enqueues and returnstask_idimmediately

Celery worker processes heavy AI workload

Frontend polls and receives live event updates

Runtime design9 / 18

9. Realtime Event Model

Lifecycle events:enqueued, started, completed, failed.

Backend publishes task events to Redis channel.

Node WebSocket service subscribes to channel and pushes to clients.

Client filters events by user identity and renders toasts/status banner.

Result: user gets immediate feedback without waiting on long HTTP round-trips.
