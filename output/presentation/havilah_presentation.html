  • Existing workflows separate notes, explanations, quiz prep, and revision.

Technical Architecture

Frontend -> FastAPI -> PostgreSQL / MongoDB / MinIO(Local)\n -> Redis -> Celery Worker\n -> Redis Pub/Sub -> Node.js WebSocket Server -> Frontend

Data Stores

PostgreSQL for relational data, MongoDB for generated content documents, Redis for queue + broker.

LLM + Async + Realtime

OpenAI models are used as the primary LLM for content planning and generation, with Anthropic support in learning asset flows. Heavy AI tasks run in Celery; progress is pushed via WebSocket events.

What "Queue Async Processing" Means

Without Queue

  • Browser waits for full AI response.
  • Risk of timeout on heavy tasks.
  • Poor UX under load.

With Queue

  • API immediately returns a task ID.
  • Celery worker processes in background.
  • Frontend gets live status updates.

Result: faster UI response and better reliability for AI-heavy operations.

Key API Endpoints

  • POST /api/v1/content/process/async - enqueue content processing task
  • GET /api/v1/content/tasks/{task_id} - fetch task status/result
  • POST /api/v1/content/from-upload - extract and generate assets from uploaded file
  • GET /api/v1/feed/videos - learning feed data
  • POST /api/v1/quiz/generate - topic-based quiz generation

Risks and Mitigations

Risk

  • External AI API latency/failure
  • Queue backlog during peak usage
  • Ephemeral local file storage in cloud

Mitigation

  • Async queues + retries + fallback handling
  • Worker scaling and queue monitoring
  • S3/MinIO backend for persistent storage
